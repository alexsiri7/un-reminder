The Un-Reminder — Privacy Policy

Last updated: 19 April 2026

The Un-Reminder is a native Android application that stores all user data exclusively on the user's device. Map tile images are fetched from OpenStreetMap tile servers when the location picker is open; these requests contain no personal data. The app also includes an optional, user-initiated feedback tool — see the "In-app feedback" section below.

Data handled on-device

  • Habits, time windows, locations, and trigger logs you create — stored in a local SQLite database on your phone.
  • Approximate and precise location — used only by the on-device geofencing service to detect arrival at locations you configure. Location data is never transmitted off the device.
  • Notification text generated by an on-device language model (Gemma 3 1B via LiteRT-LM). The model file is downloaded once on first launch and stored on your device. When using on-device generation, prompts and responses do not leave your device.

Optional cloud text generation

The app can optionally use a Cloudflare Worker backend to generate notification text via Requesty.ai (an OpenAI-compatible API gateway). When this feature is active, the following data is sent to the cloud:

  • Habit title, tags, and description/notes
  • Location name (user-defined label, not GPS coordinates)
  • Time of day

No GPS coordinates, personal identifiers, or trigger history are transmitted. The Worker does not store request data beyond transient spend counters. This feature is optional; the app falls back to on-device generation when unavailable.

In-app feedback (optional)

The app includes an optional feedback tool accessible from the Settings and Recent Triggers screens. When you choose to submit feedback, the following data is sent to GitHub.com:

  • An annotated screenshot of the current screen (may include habit names, location names, or trigger text visible at the time).
  • A description you type.

This data is stored as a public GitHub issue in the alexsiri7/un-reminder repository. Do not include sensitive personal information in feedback submissions. No feedback is sent without your explicit action.

Automated crash reporting (Sentry)

Release builds of this app use Sentry (sentry.io) to automatically capture unhandled exceptions in the AI autofill subsystem. When an error occurs, the following data is transmitted to Sentry's servers:

  • Exception type and stack trace.
  • App version and build environment ("release").
  • A component tag identifying the error source (e.g. "litert-lm-init", "ai-ui", "pool-empty").

In addition to unhandled exceptions, the app may emit non-crash diagnostic events to Sentry — for example, when the cloud notification variant pool is exhausted for a habit. These events carry the same minimal metadata (component tag, app version, build environment) and no personal data.

No personal data is attached: isSendDefaultPii = false, screenshots are not captured (isAttachScreenshot = false), UI snapshots are not captured (isAttachViewHierarchy = false), and performance tracing is disabled (tracesSampleRate = 0.0). This transmission is automatic and occurs only in release builds with a configured DSN. It is not user-initiated.

What we do not do

  • No analytics or behavioural telemetry.
  • No advertising.
  • No user account, sign-in, or cloud sync.
  • No third-party data sharing — except: (1) when you explicitly submit feedback via the in-app tool (see above), (2) automatic crash reports sent to Sentry in release builds (see above), and (3) habit metadata sent via the optional cloud text generation feature (see above).

Crash reporting

The app uses Sentry to automatically report unhandled exceptions, ANRs, and native crashes. Reports contain technical metadata including: stack traces, Android OS version, device model and screen resolution, app version, device locale, available memory, and connectivity type. Reports do not include any personal information, habit content, notification text, or location data, and are not used to identify individual users. Crash reports are sent over HTTPS and are visible only to the developer. You can review Sentry's privacy policy at sentry.io/privacy.

Permissions

The app requests the following Android runtime permissions, each used solely for the on-device features described above:

  • POST_NOTIFICATIONS — to display habit prompts.
  • ACCESS_FINE_LOCATION, ACCESS_BACKGROUND_LOCATION — to detect arrival at locations you configure.
  • SCHEDULE_EXACT_ALARM — to fire stochastic triggers at their scheduled times.
  • INTERNET — five purposes: (1) to download map tiles from OpenStreetMap when you use the location picker (no personal data sent); (2) when you explicitly submit in-app feedback, to upload your annotated screenshot and description to the app's GitHub repository as a GitHub issue (always user-initiated; nothing is sent automatically); (3) to send crash reports to Sentry (no personal information, habit content, or location data included); (4) to download the on-device LLM model file (Gemma 3 1B, ~700 MB) on first launch from the app's CDN — this occurs automatically and contains no personal data; and (5) when optional cloud text generation is enabled, to send habit metadata to the Cloudflare Worker backend (see "Optional cloud text generation" above).

Your data

Because all data is stored on-device, uninstalling the app or clearing its storage permanently removes all your data. There is no server-side copy to delete.
